# 服务地图

本仓库包含的服务清单与职责划分。**这是仓库级的第一张图**，agent 读完这一张才能定位自己在哪个服务工作。

## 字段规范

每个服务必须填全：
- `名称`：对应 Maven 模块名或根目录名
- `一句话职责`：禁止超过 30 字，禁止出现"提供各种能力""统一管理"等废话
- `代码路径`：相对仓库根的路径
- `对外暴露`：REST / MQ Topic / RPC / 无
- `对内依赖`：依赖哪些本仓库内的其他服务
- `外部依赖`：依赖哪些外部系统（数据库、外部 API）

---

## 服务清单

### service-a（示例 ✅）

- **一句话职责**：处理用户登录与会话签发
- **代码路径**：`service-a/`
- **对外暴露**：REST `POST /api/v1/auth/login`、`POST /api/v1/auth/refresh`
- **对内依赖**：service-b（调用用户档案查询接口）
- **外部依赖**：MySQL(auth_db)、Redis(session)
- **owner**：张三

### service-b（示例 ✅）

- **一句话职责**：维护用户档案与组织结构
- **代码路径**：`service-b/`
- **对外暴露**：REST `GET /api/v1/users/{id}`、MQ `user.profile.changed`
- **对内依赖**：无
- **外部依赖**：MySQL(user_db)

---

## 服务拆分的理由（必填一段）

> 为什么这两个服务是分开的，而不是一个？写下原始的决策理由。
> 这一段是给 agent 做"是否应该跨服务合并代码"这类决策时用的。

示例：
- auth 与 user 拆分的原因是登录频率远高于档案读写，需要独立扩缩容；且 auth 的故障不应影响 user 档案查询。

---

## ❌ 反例（不要这样写）

- ❌ "service-a：核心业务服务，提供各种认证能力" —— 无信息量
- ❌ "service-b：用户中心，支持用户的 CRUD、查询、授权、权限、审计等" —— 职责不清，所有东西塞一起
- ❌ 缺少代码路径 —— agent 无法定位
