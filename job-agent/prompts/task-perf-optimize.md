# 任务: 性能优化 (perf-optimize)

## 任务目标

定位性能瓶颈，实施针对性优化，确保优化有可测量的效果。

---

## 核心原则

> **先测量，后优化。没有数据支撑的优化是盲目的。**

---

## 任务执行流程

### 1. 建立基线

在优化之前，先获取当前性能数据：
```bash
# 示例：运行性能测试/基准测试
<benchmark_command> > /tmp/perf-baseline.log 2>&1
tail -20 /tmp/perf-baseline.log
```

如果项目没有现成的性能测试，先编写关键路径的基准测试。

### 2. 定位瓶颈

按优先级检查以下常见瓶颈：

| 优先级 | 瓶颈类型 | 检查方法 |
|--------|----------|----------|
| P1 | SQL 查询（N+1、全表扫描、缺索引） | 检查 MyBatis/JPA 映射和查询日志 |
| P2 | 不合理的循环（嵌套循环、重复计算） | 代码审查热路径 |
| P3 | 缺少缓存（重复查询、重复计算） | 分析方法调用频率 |
| P4 | 序列化/反序列化效率 | 检查 JSON/XML 处理 |
| P5 | 资源泄露（连接未关闭、线程泄露） | 检查 try-with-resources 使用 |
| P6 | 不合理的同步（过粗锁粒度） | 检查 synchronized/Lock 使用 |

### 3. 输出优化决策

```
## 决策: 优化 <目标>
- 瓶颈: <定位到的瓶颈描述>
- 数据: <基线性能数据>
- 方案: <优化手法>
- 预期: <预期改善幅度>
```

### 4. 实施优化

- 每轮只做**一个**优化点
- 优化后运行全量测试确保功能不受影响
- 重新运行性能测试获取优化后数据

### 5. 验证效果

对比基线数据和优化后数据：
```
## 优化效果
- 优化前: <基线数据>
- 优化后: <新数据>
- 改善: <改善百分比>
```

如果优化没有明显效果（<5% 改善），回滚修改。

---

## 注意事项

- 性能优化使用 `perf(<范围>): <描述>` 作为 commit 类型
- 不要以牺牲代码可读性为代价做微优化
- 如果瓶颈在数据库层面，考虑添加索引而非修改代码逻辑
- 缓存优化必须考虑缓存一致性和过期策略
