# 文档治理规则

代码的价值不仅在于可执行，更在于可理解。你必须在每个版本发布时同步维护项目文档。

---

## CHANGELOG.md

每个版本发布后，必须在项目根目录的 `CHANGELOG.md` 文件中记录变更，格式参考 [Keep a Changelog](https://keepachangelog.com/)：

```markdown
# Changelog

## [0.3.0] - 2026-04-23

### Added
- 新增 XXX 功能

### Changed
- 优化 YYY 逻辑

### Fixed
- 修复 ZZZ 问题

## [0.2.0] - 2026-04-22
...
```

### 规则
- 基于本次版本中的 git commit 记录生成
- `feat` → Added，`fix` → Fixed，`refactor`/`perf` → Changed，`docs` → 不记录
- 使用简体中文描述
- 最多保留最新 10 个版本的记录

---

## 架构决策记录 (ADR)

当你做出以下类型的决策时，**必须**在 `docs/adr/` 目录下创建 ADR 文件：

- 引入新的外部依赖
- 修改模块间的依赖方向
- 改变数据存储方案
- 改变 API 接口契约
- 大规模重构（涉及 3 个以上文件的结构性修改）

### ADR 文件命名
`NNNN-decision-title.md`，其中 NNNN 从已有 ADR 文件推断编号（首个为 0001）。

### ADR 模板
```markdown
# NNNN: <决策标题>

**日期**: <YYYY-MM-DD>
**状态**: 已接受

## 背景
<为什么需要做这个决策？问题是什么？>

## 决策
<选择了什么方案？>

## 备选方案
<考虑过哪些其他方案？为什么没有选择？>

## 影响
<这个决策的正面和负面后果是什么？>
```

---

## README.md

每个版本发布后更新 `README.md`，确保包含：
- 项目简介和功能概述
- 最新版本号和变更亮点
- 快速开始指南
- 构建和测试命令

最多保留最新 5 个版本的介绍，旧版本合并压缩。

---

## 注意事项

- 文档使用简体中文
- 文档更新的 commit 使用 `docs(<范围>): <描述>` 格式
- 文档不是"做完代码后的附加品"，它是版本发布的**必要条件**
