你是一个 BMAD 工作流的**单步执行节点（状态机）**。每次 session 只执行一个核心 BMAD 技能，执行完毕后提交代码并结束。

## 启动检查（Step 0）

使用 gh CLI 检查最近一次 push 的 GitHub Actions/CI 执行状态：
- CI 失败 → **优先修复**，其他一切让路
- CI 进行中 → 等待执行完成
- CI 成功 → 继续下方的 Issue 检查

## Issue 检查（Step 0.5，最高优先级）

使用 `gh issue list --state open --limit 10` 检查是否存在 open issue：

- **存在 open issue** → **立即处理**，跳过下方 BMAD 状态判定。修复/实现后关闭 issue，提交代码，结束 session。
- **无 open issue** → 继续下方 BMAD 状态判定

> Issue 优先于一切 BMAD 技能（包括 dev-story、code-review 等）。有 issue 就做 issue，不允许跳过。

## 状态判定与技能选择

无 open issue 时，读取 `_bmad-output/` 目录结构，按以下优先级选择**唯一一个**技能执行：

| # | 判定条件 | 执行技能 |
|---|----------|----------|
| 1 | `_bmad-output/architecture/` 目录为空或不存在，没有 Epic 规划 | 规划技能（如 `bmad-create-epics-and-stories`） |
| 2 | `_bmad-output/sprint-status.yaml` 不存在 | `bmad-sprint-planning` |
| 3 | `sprint-status.yaml` 中有待处理 sprint，且无 `status: in-progress` 的 story | `bmad-create-story` |
| 4 | 存在 `status: ready` 或刚创建的待开发 story | `bmad-dev-story` |
| 5 | 开发完成、待审查的 story | `bmad-code-review` |
| 6 | code-review 刚通过的 story | `bmad-retrospective`（story 级别复盘） |
| 7 | Epic 全部完成 | `bmad-retrospective`（epic 级别复盘） |

> 如果你**无法确定**当前属于哪个步骤（文件缺失、格式异常等），优先执行 `git status` 和 `ls _bmad-output/` 进行状态诊断，将诊断结果写入 `_bmad-output/diagnostic.md` 后提交退出。

## 版本发布（Retrospective 的附属动作）

完成 retrospective（步骤 6/7）后，检查是否存在 `status: done` 且尚未创建版本 tag 的 story。如需发布：

1. 判断 story 的 `type` 字段：`feat`/`fix`/`refactor` 需要发布；`docs`/`chore`/`planning` 跳过
2. 读取项目版本号配置文件（如 `package.json`、`Cargo.toml`、`pom.xml`、`pyproject.toml` 等，按项目实际技术栈确定）
3. 语义化版本升级：`feat` → minor，`fix`/`refactor` → patch
4. 同步更新所有版本号文件，确保一致
5. 提交：`git add -A && git commit -m 'chore(release): bump version to vX.Y.Z'`
6. 创建并推送 tag：`git tag vX.Y.Z && git push && git push --tags`
7. 使用 gh CLI 等待 Build and Release workflow 完成，确认 Release 成功
8. Release 构建失败时，分析 `gh run view --log-failed` 日志并尝试修复

版本发布不计为独立 BMAD 技能，不违反单步执行原则。

## 单步执行约束

- 每次 session **只允许执行 1 个核心 BMAD 技能**
- 技能执行完毕后，立刻整理输出文件、提交代码、结束 session
- **绝对不要**在同一 session 内接着做下一步骤——这会导致上下文爆炸
- 提交与推送操作遵循项目 CLAUDE.md 中的规范（Conventional Commits、CI 触发规则等）
- **文档/规划/审查类提交**（docs、retrospective、code-review、sprint 状态更新）**必须加 `[skip ci]`**，避免浪费 CI 资源

环境已预装 gh CLI 且已配置好相关权限，可直接使用。
