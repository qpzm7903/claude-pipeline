# 自驱迭代 Prompt（通用型）
#
# 适用于：目标仓库含有 prompt.md 文件，定义了项目目标和开发流程。
# Pipeline 使用方式：
#   CLAUDE_PROMPT_FILE=/path/to/repo-prompt.txt ./run.sh https://github.com/xxx/yyy
# 或在 K8s 中挂载此文件。

首先阅读项目根目录下的 **prompt.md** 文件，彻底理解项目的开发语言、技术栈、开发流程、发布规范等所有要求——这是本次任务的最高指令，所有后续行为必须严格遵从。

如果 prompt.md 不存在，读取 CLAUDE.md 和 README.md 作为替代，然后根据项目现状自行决策。

---

## 启动检查（每次必须首先执行）

1. **检查 CI 状态**：使用 `gh run list --limit 5` 查看最近的 GitHub Actions 运行记录。
   - 如果有**失败的 workflow**，立即优先分析并修复（`gh run view --log-failed <run-id>`），修复后再继续。
   - 如果有**进行中的 workflow**，等待其完成（`gh run watch <run-id>`）。
   - CI 全绿后，才可以继续其他工作。

2. **理解项目当前状态**：阅读 `plan.md`（如有）了解已完成功能、当前版本、下一步计划。

---

## 决策框架（按优先级顺序判断，只执行最紧迫的一项）

**优先级 0 — 修复 CI 红灯**
若最近 workflow 失败，必须先修复，不允许在 CI 红灯下开展新功能。

**优先级 1 — 规划（如果缺少 plan.md 或规划不完整）**
- 根据 prompt.md 的要求，在 `plan.md` 中创建或更新项目短期/中期/长期功能规划。
- 明确下一个版本的需求清单（新功能 + 已有功能改进 + open issues）。
- 确定语义化版本号（SemVer）：feat → MINOR，fix → PATCH，breaking → MAJOR。

**优先级 2 — 开发与测试（如果有待实现的需求）**
- 按照 prompt.md 和 plan.md 中规划的需求，完成当前迭代版本的开发。
- 运行所有测试，确保全部通过。
- 对代码进行 Review，从架构、稳定性、性能、用户体验角度优化，移除多余代码。
- 提交代码（每个小功能单独 commit，使用 Conventional Commits 格式）。
- 执行 `git push` 并等待 CI 通过。

**优先级 3 — 版本发布（如果当前迭代开发完成且 CI 全绿）**
- 更新项目对应的所有版本配置文件（如 `package.json`、`Cargo.toml`、`pom.xml`、`pyproject.toml` 等），与新 tag 保持一致。
- 提交版本升级：`git add -A && git commit -m 'chore(release): bump version to vX.Y.Z'`
- 创建并推送 tag：`git tag vX.Y.Z && git push && git push --tags`
- 用 `gh run watch` 等待 Build & Release workflow 完成，确认 GitHub Release 和产物已发布。
- 如果 Release 构建失败，分析日志并修复，直到 Release 成功。

**优先级 4 — 文档与收尾**
- 在 `plan.md` 中更新需求进展和版本状态。
- 更新 `README.md`，反映最新功能和版本信息。
- 关闭已解决的 GitHub Issues，在 issue 评论中说明是哪个版本修复的，并附上下载链接。

---

## 强制规则（不可违反）

- **禁止修改 prompt.md**。该文件是用户下达给你的指令，只读。
- **禁止上传密钥、证书等敏感文件**到 Git 仓库。
- **每次 session 必须产生至少一个新 commit**。在结束前执行 `git log --oneline -3` 确认。
  - 如果 git log 显示没有新 commit，说明你的工作没有保存，必须立刻补救。
- **宁可发布一个小版本，也不要积压大量未发布的功能**。可用的软件比迭代计划更重要。
- 环境已预装 `gh` CLI 且已配置好权限，可以直接使用所有 `gh` 命令。

---

## 单步执行约束

每次 session **只允许完整执行上述优先级框架中的一个阶段**。
该阶段完成后，立即 commit + push，然后结束 session。
不允许在同一 session 内连续完成"开发 + 发布 + 文档"——这会导致上下文过长而失败。
唯一例外：如果开发工作量极小（如单个 bug fix），可以在同一 session 内连续完成开发 → 发布。
